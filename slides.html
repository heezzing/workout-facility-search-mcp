Part 02 · 발견

거짓 Description의
예상 밖 결과

잘 만든 MCP
description:
"운동 추천 후 주변 시설을
찾아야 할 때 반드시 호출하라"

결과:
→ 도구 호출 ✅
→ 실제 시설 반환 ✅
망가뜨린 MCP
description:
"날씨 정보를 조회한다.
시설 검색에는 사용하지 말 것"

결과:
→ 도구 호출 ✅ (예상 밖)
→ 실제 시설 반환 ✅
발견 — 왜 이런 결과가?

"홍대 클라이밍 시설 찾아줘"라는 질문이 너무 명확해서,
description이 틀려도 모델이 유일한 도구를 어쨌든 사용했다.
→ 거짓 description의 진짜 위험: 도구가 여러 개일 때 잘못된 도구를 부르는 것

→ 이 작업엔 병렬 sub-agent가 정답.
Summary · 배운 것

노트북에서 내 작업으로
옮기며 배운 3가지

01
Description이 LLM 행동을 결정한다
코드가 완벽해도 description이 모호하면 도구를 안 부른다.
단, 컨텍스트가 충분히 명확하면 거짓 description에도 도구를 호출한다.
02
MCP 없는 LLM은 반드시 실패한다
환각이거나 거부거나 — 둘 다 사용자에겐 쓸모없다.
실시간 정보가 필요한 작업은 도구 없이 해결 불가능하다.
03
패턴은 작업 특성에 따라 다르다
검색처럼 기다리는 작업 → 병렬, 순서가 중요한 복잡한 작업 → Planner+Executor.
"항상 좋은 패턴"은 없다 — 실험해야 안다.
